# MyGym Intake Protocol

Use this protocol for turning messy inputs into structured project work.

Status: active under Developer Workflow V2 adopted on 2026-04-05.

This protocol sits underneath the higher-level workflow in `DEV_WORKFLOW.md`.
Apple Notes is the default raw capture inbox. This file explains how raw MyGym inputs get classified and routed once they enter the project workflow.

## Accepted raw inputs
- voice-note summaries
- “save this” messages
- feature ideas
- bugs and frustrations
- screenshots / observations
- market thoughts
- onboarding concerns
- code concerns

## Triage categories
Each new input should be classified as one or more of:
- Feature
- Bug
- UX issue
- GTM insight
- Positioning insight
- Technical debt
- Decision
- Research question

## Tag mapping
Use these working tags:
- `code`
- `product`
- `ux`
- `gtm`
- `ops`
- `research`

## Routing rules
- backlog-worthy work → `BACKLOG.md`
- durable project truth → `PROJECT_BRIEF.md`
- market / positioning insight → `GTM.md`
- stable feature understanding → `FEATURE_INVENTORY.md`
- meaningful decisions → `DECISIONS.md`
- implementation quality / release concerns → `CODE_READINESS.md`

## Prioritization lanes
- NOW
- NEXT
- LATER
- PARKING LOT

## Working modes
### Builder mode
Use for:
- code review
- implementation
- refactors
- debugging
- release prep

### Founder mode
Use for:
- positioning
- messaging
- onboarding clarity
- launch sequencing
- monetization
- user value framing

## Principle
Capture first, structure second, execute third.
